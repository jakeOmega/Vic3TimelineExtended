## Focus ranking

Prioritize, in this order:

1. **Internal logic** — triggers fire when intended, effects do what's described, no orphan references, scope chains valid, modifier durations in correct units (days vs months — note `*_modifier_time` constants are days), state-scoped scaling runs from `on_yearly_pulse_state`.
2. **Mod convention adherence** — patterns documented in `CLAUDE.md` § Building blocks and `docs/guides/scripting_best_practices.md`. **If you see repeated convention violations across files, flag it as a question for the user** rather than fixing each instance — the convention may need to broaden.
3. **Balance / design coherence** — within-system consistency, ludo-narrative consistency (e.g. becoming an exploited dependency shouldn't raise SoL), modifier magnitudes out of whack vs peers.
4. **Localization narrative / style** — tone consistency (`event_creation_guide.md` § General Tone), mechanical-narrative alignment.

Explicitly **deprioritized** (don't actively search — these are covered by procedural audits running on `/reload`):

- Syntax / parser correctness (the engine and `paradox_file_parser.py` flag these)
- Loc completeness (`loc_coverage_audit`)

## Bar for filing

Every finding must include:

- A specific `file:line` reference.
- A concrete proposed fix **or** a specific question to the user.
- Reasoning tied to a doc, a convention, or a vanilla mechanic (cite the source).

"This section feels underspecified" without a concrete proposal does not get filed.

## Auto-fix rule

> If fixing it is no more effort than filing an issue, fix it directly via PR. Otherwise, file an issue.

Judgment-based, not category-based. Calibrating examples:

- Typo in a loc string → fix.
- Missing `# REVIEWED YYYY-MM-DD` on a known-intentional suppression → fix.
- Two effects firing in the wrong order per a clear doc rule → fix.
- A modifier that's wrong by 30× because months were used in place of days → fix.
- A whole system of events whose IG approvals don't match the politics reference → **issue** (the system may need re-thinking, not 12 patches).
- Anything where the fix depends on understanding designer intent → **issue**.

**Cap**: 5 files / 50 lines changed per night across all auto-fix PRs. If more would fit, fix some and file an issue listing the rest.

**Branch + PR convention**:

- Branch: `nightly-audit/2026-06-06-<short-slug>` from `main`.
- PR title: `nightly-audit(2026-06-06): <short description>`.
- PR body: cite the auto-fix rule, link to this night's `docs/audits/nightly/2026-06-06/` directory, list the rule each fix passes.

## Fast verify after fixes

After each batch of mod-file edits (and before opening the PR):

1. `curl -X POST 'http://localhost:8950/reload?mod_only=true&audits_only=true'` — fast (~25s vs ~90s for a full reload); reuses the cached vanilla parse and skips the file-rewriting generators. The only on-disk side effect is updates to `docs/engine/*_report.md` reflecting the new warnings.
2. Inspect the response's `warnings` array. If a warning references one of your edits, revert that hunk.
3. Iterate until the warnings are clean (or attributable to pre-existing issues unrelated to your edits — note those in the PR body if relevant).

Do **not** use the unflagged `POST /reload` for verification — it re-runs the file-rewriting post-load generators and will dirty your working tree with their output.

## Dedup workflow

3. If a match exists with roughly the same finding: **comment on the existing issue, don't file a new one**.

## Wrap-up

After all targets are audited:

1. Update `docs/audits/.nightly_coverage.json` **via the helper script** — never hand-edit it (#166):
   ```bash
   echo '{"path/to/file_a.txt": 2, "path/to/file_b.txt": 0}' | \
     python3 scripts/nightly_audit_state_update.py \
       --targets-json docs/audits/nightly/2026-06-06/targets.json \
       --findings-json - --date 2026-06-06
   ```
   The helper bumps `audit_count`, stamps `last_audited = 2026-06-06`, sets `recent_findings` from the map you pass (omit a path → 0), appends slice bounds to `partial_coverage` for sliced targets, and decays other entries older than 90 days. Hand-edits / `Edit`-tool appends silently lose data the next time the file is loaded — always go through the helper.
2. **State bump rides in a PR. Never direct-push to `main`.**
   - If there are auto-fix PRs this night: include the `.nightly_coverage.json` diff in the **first** auto-fix PR (or a dedicated coordinating PR — one PR per night carries the state diff).
   - If there are no auto-fix PRs: open a state-bump-only PR titled `nightly-audit(2026-06-06): state bump` from branch `nightly-audit/2026-06-06-state-bump`. Enable auto-merge if available; otherwise leave open for manual merge.
